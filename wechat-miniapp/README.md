# 微信小程序方向：八字关系签

本目录用于沉淀微信小程序版本的产品设计、交互方案、数据契约与后续实现骨架。

## 当前定位

「八字关系签」不是传统合盘工具，而是一个基于八字命理的暧昧关系邀请产品。

核心动作有两条：

> 落签给 TA：A 输入自己的八字，生成一支只给某个人打开的关系签；B 收到后输入自己的八字，看到一份只属于 B 的双方关系推论。A 能看到 B 的身份与打开/生成状态，但看不到 B 的内容。

> 向 TA 求签：A 发起邀约，请 B 留下一支密封签；B 填写的八字 A 看不到。B 留签后，A 输入自己的八字，看到一份只属于 A 的双方关系推论。

## 关键约束

- 八字必须站到前台，这是产品独特性来源。
- 首发场景聚焦暧昧/恋爱，不做泛关系平台。
- 第一版允许 A 发给多人，A 能看到有多少人试了；B 不知道 A 发给了多少人。
- B 的内容默认私密，A 能看到 B 的身份与状态，但看不到 B 的结果。
- 求签动线中，B 的八字密封保存，A 只能看到自己的关系视角。
- 输出必须基于命理专业判断，不强行暧昧、不制造命运判决。
- 不输出“你们不合”“TA 不爱你”“一定没结果”等伤害性结论。
- 首版不做双向解锁，只保留 B 的一句话回馈。

## 现有项目复用原则

现有项目的核心计算能力已经分层：

- `engine/`：Python 排盘与规则分析引擎。
- `report/`：正式报告层与 prompt/template 真相源。
- `web/backend/app/api/engine.py`：公开的轻量排盘 API。
- `app/src/engine/*`：旧版 TypeScript 引擎多处标记 deprecated，仅可参考，不作为新增逻辑来源。

微信小程序应复用 Python 引擎与后端 API，不复制旧 TS 合婚逻辑。
